How does it work?

HTTP is made up of headers and bodies. Both on the request and the response side of things.

HTTP is actually a super simple protocol. Its probably why its been around for so long. Generally what people think of HTTP as is the current version, HTTP/1.1, HTTP/1.0 still works, and 0.9 is very hit or miss When your browser starts talking to a server, thats a request, and when it responses its a ... response. Yea I know. The cool thing about HTTP is it really allows for a lot of expandability. You really only need the first line of the request to do anything.

Request

The request is made up of a method, a path and a protocol version. In this case, the method is GET, the path is slash index and the version is 1.1.

Then it is followed buy any number of key value pairs, separated by a colon, which make up the headers.

Then finally a blank line and any bodies.

Response

The response follows the same sort of pattern. Http version, followed by status code and status string.
Then headers.
Blank line
Then body. Usually html page, but could be json or binary or anything else.

Types of HTTP Requests (1.0 - Simplified)

  • Getting Data
  • Should not change state
  • Should be repeatable
  • Just returns header, no body
  • Not really going to cover
  • Sending Data
  • Modifies state
  • No guarantees if sending again will do the same thing

HTTP/1.1 Added More

  • OPTIONS
  • PUT
  • DELETE
  • TRACE
  • CONNECT
  • Idempotent
  • "Save"
  • "Store at this location"
  • Idempotent
  • Delete this item (if exists)
Really a HTTP Verb can be anything. These are just standardized behavior. The only really common ones from this list is PUT and DELETE. Both should be idempotent, meaning it should have the same effect if you repeat the request. Like most things HTTP, this is just a guideline.

"Best" Practices

There's no right or wrong way to do rest. As long as things are valid http, then its probably fine. But as a developer that spends a lot of time integrating with other apis, there are some things I've learned.

URL Schemas

  • GET /resource - should return all
  • GET /resource/id - should return one

  • POST /resource - should add one

  • PUT/POST /resource/id - should update/store one

  • DELETE /resource/id - should delete one
REST resources generally follow a practice like files on your disk
.
GET would be the same as load. If you get a folder or directory, you get a listing of files
. If you GET a specific file, you get the contents
.
POST and PUT is kinda where that analogy falls apart, but it still is similar. You want to save a new file, Its a POST to a folder. If you want to overwrite, its PUT to a file
.
DELETE is simplest, it'll just delete what you tell it to. Generally REST interfaces just stick with deleting files.

Response Code

  • 200 - All good
  • 201 - Created Successfully
  • 301 - Always look here instead
  • 302 - Look here instead
  • 401 - Security pls. Who are you?
  • 402 - Not allowed
  • 404 - Not found
  • 418 - I'm a little tea pot
  • 500 - PANIC ERROR HAS HAPPENED

https://en.wikipedia.org/wiki/List_of_HTTP_status_codes
There are many many response codes.

I've tried to list a few that I see pretty often, these are not the official titles, just sorta my fun way of remembering them.

They are segmented into blocks.
100s are informational, not often seen unless you are writing a server like apache or nginx
200s are various forms of good
300s are redirections
400s are errors (expected)
500s are server errors (bad)

Yes, there's a http code for being a little tea pot. Not making it up.
